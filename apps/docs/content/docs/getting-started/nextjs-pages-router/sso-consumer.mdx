# Sign in with Google, GitHub, and OIDC

This guide walks you through adding OAuth 2.0 and OpenID Connect as a Relying Party (consumer) in your Next.js Pages Router project. Your application will redirect users to an external identity provider (Google, GitHub, Auth0, Keycloak, or any OIDC-compliant provider), then receive an authorization code and exchange it for user profile data.

<Callout type="info">
  This is the **consumer** side — your app is the Relying Party. If you want to **become** an OIDC provider and issue tokens to other applications, see [Become an OIDC provider](/getting-started/nextjs-pages-router/sso-provider).
</Callout>

<Callout type="info">
  This guide assumes you have completed the [core Next.js Pages Router setup](/getting-started/nextjs-pages-router).
</Callout>

SSO consumer functionality is built into `@holeauth/core/sso` — no separate plugin installation is required.

<Steps>
<Step>
### Configure providers

The `createPagesAuthHandler` function accepts a `providers` array. Each provider is created by calling a factory function exported from `@holeauth/core/sso`.

<Tabs items={['Google', 'GitHub', 'Discord', 'Microsoft', 'Generic OIDC']}>
  <Tab value="Google">
```ts title="lib/auth.ts" {2,4-10,18}
import { createPagesAuthHandler } from '@holeauth/nextjs-pages-router';
import { GoogleProvider } from '@holeauth/core/sso';
import { createHoleauthAdapters } from '@holeauth/adapter-drizzle/pg';
import { db } from '@/db/client';
import { holeauth } from '@/db/schema';

const providers = [
  GoogleProvider({
    clientId: process.env.GOOGLE_CLIENT_ID!,
    clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    redirectUri: `${process.env.APP_URL}/api/auth/callback/google`,
  }),
];

const adapters = createHoleauthAdapters({ db, tables: holeauth.tables });

export const auth = createPagesAuthHandler({
  secrets: { jwtSecret: process.env.HOLEAUTH_SECRET! },
  adapters: {
    user: adapters.user,
    session: adapters.session,
    account: adapters.account,
    auditLog: adapters.auditLog,
    verificationToken: adapters.verificationToken,
    transaction: adapters.transaction,
  },
  providers,
  tokens: {
    accessTtl: 60 * 15,
    refreshTtl: 60 * 60 * 24 * 30,
    cookiePrefix: 'holeauth',
    sameSite: 'lax',
  },
  registration: {
    selfServe: true,
    inviteTtlSeconds: 7 * 24 * 60 * 60,
    inviteUrl: ({ token }) =>
      `${process.env.APP_URL}/register/accept?token=${encodeURIComponent(token)}`,
  },
  allowDangerousEmailAccountLinking: false,
});
```

<Step>
### Create a sign-in button

Add direct links to the SSO authorization endpoints:

```tsx title="pages/login.tsx"
export default function LoginPage() {
  return (
    <div>
      <h1>Sign in</h1>
      <a href="/api/auth/sso/google/authorize">Sign in with Google</a>
      <a href="/api/auth/sso/github/authorize">Sign in with GitHub</a>
    </div>
  );
}
```

The `authorize` endpoint generates the OAuth 2.0 authorization URL, stores a CSRF state parameter in a cookie, and redirects the user to the provider.

To customize the redirect after sign-in, add a `redirectTo` query parameter:

```tsx
<a href="/api/auth/sso/google/authorize?redirectTo=/dashboard">Sign in with Google</a>
```
</Step>

<Step>
### Handle the callback

Holeauth automatically registers a callback route at `/api/auth/callback/:provider` when you mount the API route handler. The callback:

1. Validates the `state` parameter against the CSRF cookie.
2. Exchanges the authorization code for an access token.
3. Fetches the user profile from the provider.
4. Creates or links a user account in your database.
5. Issues holeauth session tokens and redirects the user to your app.

By default, the callback redirects to `/` on success.
</Step>

<Step>
### Account linking

When a user signs in via SSO and an account with the same email already exists, holeauth checks the `allowDangerousEmailAccountLinking` flag:

- If `false` (default), the sign-in fails with a `LINKING_FORBIDDEN` error.
- If `true`, the SSO account is linked to the existing user.

<Callout type="warn">
  Setting `allowDangerousEmailAccountLinking: true` allows anyone who can verify an email address (via a malicious OAuth provider) to hijack an existing account. Only enable this if you trust all configured providers.
</Callout>

For manual linking control, implement a server action that checks the user session:

```ts title="pages/api/link-account.ts"
import type { NextApiRequest, NextApiResponse } from 'next';
import { getServerSidePropsSession } from '@holeauth/nextjs-pages-router';
import { auth } from '@/lib/auth';

export default async function handler(req: NextApiRequest, res: NextApiResponse) {
  if (req.method !== 'POST') {
    return res.status(405).json({ error: 'Method not allowed' });
  }

  const session = await getServerSidePropsSession({ req } as never, auth.config);
  if (!session) {
    return res.status(401).json({ error: 'Unauthorized' });
  }

  const { provider, token } = req.body;
  // Implement your linking logic here
  return res.status(200).json({ ok: true });
}
```

## How it works

SSO sign-in follows the OAuth 2.0 Authorization Code flow with PKCE (for public clients) or client secret (for confidential clients):

1. User clicks the sign-in button.
2. Your app redirects to the provider authorization endpoint with a CSRF state token.
3. Provider authenticates the user and redirects back to your callback URL with an authorization code.
4. Your app exchanges the code for an access token.
5. Your app fetches the user profile and creates or links a user account.
6. Your app issues holeauth session tokens.

<Callout type="info">
  Holeauth stores the provider account ID, access token, and refresh token (if available) in the `accounts` table. This allows you to call the provider API on behalf of the user later.
</Callout>
