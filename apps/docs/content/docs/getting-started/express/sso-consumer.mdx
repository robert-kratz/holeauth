# Sign in with Google, GitHub, and OIDC

This guide walks you through adding OAuth 2.0 and OpenID Connect as a Relying Party (consumer) in your Express application. Your application will redirect users to an external identity provider (Google, GitHub, Auth0, Keycloak, or any OIDC-compliant provider), then receive an authorization code and exchange it for user profile data.

<Callout type="info">
  This is the **consumer** side — your app is the Relying Party. If you want to **become** an OIDC provider and issue tokens to other applications, see [Become an OIDC provider](/getting-started/express/sso-provider).
</Callout>

<Callout type="info">
  This guide assumes you have completed the [core Express setup](/getting-started/express).
</Callout>

SSO consumer functionality is built into `@holeauth/core/sso` — no separate plugin installation is required.

<Steps>
<Step>
### Configure providers

The `createExpressAuth` function accepts a `providers` array. Each provider is created by calling a factory function exported from `@holeauth/core/sso`.

<Tabs items={['Google', 'GitHub', 'Discord', 'Microsoft', 'Generic OIDC']}>
  <Tab value="Google">
```ts title="lib/auth.ts" {2,4-7,15}
import { createExpressAuth } from '@holeauth/express';
import { GoogleProvider } from '@holeauth/core/sso';
import { createHoleauthAdapters } from '@holeauth/adapter-drizzle/pg';
import { db } from '../db/client';
import { holeauth } from '../db/schema';

const providers = [
  GoogleProvider({
    clientId: process.env.GOOGLE_CLIENT_ID!,
    clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    redirectUri: `${process.env.APP_URL}/api/auth/callback/google`,
  }),
];

const adapters = createHoleauthAdapters({ db, tables: holeauth.tables });

export const auth = createExpressAuth({
  secrets: { jwtSecret: process.env.HOLEAUTH_SECRET! },
  adapters: {
    user: adapters.user,
    session: adapters.session,
    account: adapters.account,
    auditLog: adapters.auditLog,
    verificationToken: adapters.verificationToken,
    transaction: adapters.transaction,
  },
  providers,
  tokens: {
    accessTtl: 60 * 15,
    refreshTtl: 60 * 60 * 24 * 30,
    cookiePrefix: 'holeauth',
    sameSite: 'lax',
  },
  registration: {
    selfServe: true,
    inviteTtlSeconds: 7 * 24 * 60 * 60,
    inviteUrl: ({ token }) =>
      `${process.env.APP_URL}/register/accept?token=${encodeURIComponent(token)}`,
  },
  allowDangerousEmailAccountLinking: false,
});
```

<Step>
### Client integration

To initiate an SSO sign-in flow, redirect the user to the provider login endpoint:

```html
<a href="/api/auth/login/google">Sign in with Google</a>
<a href="/api/auth/login/github">Sign in with GitHub</a>
```

Or from JavaScript:

```ts
window.location.href = '/api/auth/login/google';
```

The flow:

1. User clicks the link.
2. Server redirects to the provider (e.g. `https://accounts.google.com/o/oauth2/v2/auth?...`).
3. User authenticates and consents at the provider.
4. Provider redirects back to your callback URL (e.g. `/api/auth/callback/google?code=...`).
5. Holeauth exchanges the code for tokens, fetches the user profile, and creates or links the account.
6. Holeauth sets session cookies and redirects to your app (default: `/`).

<Callout type="info">
  To customize the post-login redirect, add a `returnTo` query parameter: `/api/auth/login/google?returnTo=/dashboard`. The server will redirect there after successful authentication.
</Callout>

## How it works

When a provider is configured:

1. Holeauth fetches the provider OIDC discovery document (for OIDC providers) or uses built-in endpoints (for Google, GitHub, etc.).
2. On `/api/auth/login/:providerId`, holeauth generates a random `state` parameter (CSRF protection), stores it in an `httpOnly` cookie, and redirects the user to the provider authorization endpoint.
3. The provider authenticates the user and redirects back to `/api/auth/callback/:providerId?code=...&state=...`.
4. Holeauth validates the `state` parameter, exchanges the `code` for an access token, fetches the user profile, and checks the `account` adapter for an existing SSO link.
5. If the account exists, holeauth issues session tokens. If not, holeauth creates a new user and links the account.
6. Session cookies are set, and the user is redirected to the `returnTo` URL or `/`.

See [SSO](/sso) for details on account linking and email verification.

## Account linking

By default, holeauth links SSO accounts to existing users by email. If a user with the same email already exists, the SSO account is linked to that user. To disable this (and require explicit account linking), set `allowDangerousEmailAccountLinking: false` in your auth configuration.

See [Account Linking](/sso/consumer) for details and security considerations.
