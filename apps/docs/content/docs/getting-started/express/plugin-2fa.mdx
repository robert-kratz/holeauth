# Two-factor authentication

This guide walks you through adding TOTP-based two-factor authentication to your Express application. Users will enroll by scanning a QR code with an authenticator app, then verify a six-digit code on every sign-in. Recovery codes are generated once during enrollment for account recovery.

<Callout type="info">
  This guide assumes you have completed the [core Express setup](/getting-started/express).
</Callout>

### Extend your database schema

Import the 2FA table factory and add it to your schema:

<Tabs items={['Postgres', 'MySQL', 'SQLite']}>
  <Tab value="Postgres">
```ts title="db/schema.ts" {3,13-14}
import { pgTable, text, timestamp } from 'drizzle-orm/pg-core';
import { createHoleauthTables } from '@holeauth/adapter-drizzle/pg';
import { createTwoFactorTables } from '@holeauth/2fa-drizzle/pg';

export const users = pgTable('user', {
  id: text('id').primaryKey().$defaultFn(() => crypto.randomUUID()),
  email: text('email').notNull().unique(),
  passwordHash: text('password_hash'),
  createdAt: timestamp('created_at', { withTimezone: true, mode: 'date' }).notNull().defaultNow(),
});

export const holeauth = createHoleauthTables({ usersTable: users });
export const twoFa = createTwoFactorTables({ usersTable: users });
```

<Step>
### Wire the plugin into your auth instance

<Tabs items={['Postgres', 'MySQL', 'SQLite']}>
  <Tab value="Postgres">
```ts title="lib/auth.ts" {4,5,9,23-28}
import { createExpressAuth } from '@holeauth/express';
import { createHoleauthAdapters } from '@holeauth/adapter-drizzle/pg';
import { db } from '../db/client';
import { holeauth, twoFa } from '../db/schema';
import { twofa } from '@holeauth/plugin-2fa';
import { createTwoFactorAdapter } from '@holeauth/2fa-drizzle/pg';

const adapters = createHoleauthAdapters({ db, tables: holeauth.tables });
const twoFactorAdapter = createTwoFactorAdapter({ db, tables: twoFa.tables });

export const auth = createExpressAuth({
  secrets: {
    jwtSecret: process.env.HOLEAUTH_SECRET!,
  },
  adapters: {
    user: adapters.user,
    session: adapters.session,
    account: adapters.account,
    auditLog: adapters.auditLog,
    verificationToken: adapters.verificationToken,
    transaction: adapters.transaction,
  },
  plugins: [
    twofa({
      adapter: twoFactorAdapter,
      issuer: 'MyApp',
      recoveryCodeCount: 10,
    }),
  ],
  tokens: {
    accessTtl: 60 * 15,
    refreshTtl: 60 * 60 * 24 * 30,
    cookiePrefix: 'holeauth',
    sameSite: 'lax',
  },
  registration: {
    selfServe: true,
    inviteTtlSeconds: 7 * 24 * 60 * 60,
    inviteUrl: ({ token }) =>
      `${process.env.APP_URL}/register/accept?token=${encodeURIComponent(token)}`,
  },
  allowDangerousEmailAccountLinking: false,
});
```

<Step>
### Available routes

The 2FA plugin registers the following routes on the holeauth router:

<TypeTable
  type={{
    'POST /api/auth/2fa/setup': {
      description: 'Generate a TOTP secret, recovery codes, and a QR code data URL for the authenticated user. Requires authentication and CSRF token.',
      type: 'endpoint',
    },
    'POST /api/auth/2fa/activate': {
      description: 'Verify a TOTP code from the user authenticator app and enable 2FA. Requires authentication and CSRF token. Body: `{ code: string }`.',
      type: 'endpoint',
    },
    'POST /api/auth/2fa/verify': {
      description: 'Complete a pending sign-in by verifying the TOTP code or a recovery code. Consumes the pending token cookie and returns session tokens. Body: `{ code: string }`.',
      type: 'endpoint',
    },
    'POST /api/auth/2fa/disable': {
      description: 'Disable 2FA for the authenticated user by verifying a TOTP code. Requires authentication and CSRF token. Body: `{ code: string }`.',
      type: 'endpoint',
    },
  }}
/>

<Callout type="warn">
  Recovery codes are single-use. Once consumed, they are removed from the database. Store them hashed, never in plain text.
</Callout>
</Step>

<Step>
### Client integration

Your client (web, mobile, or desktop) must call these endpoints. A typical enrollment flow:

1. User signs in and navigates to account settings.
2. Client calls `POST /api/auth/2fa/setup` (send CSRF token in `x-csrf-token` header or `csrfToken` body field).
3. Server returns `{ ok: true, secret, otpauthUrl, qrDataUrl, recoveryCodes }`.
4. Client renders the QR code (e.g. `<img src={qrDataUrl} />`) and displays recovery codes for the user to save.
5. User scans the QR code with an authenticator app.
6. Client prompts for a six-digit code, then calls `POST /api/auth/2fa/activate` with `{ code }`.
7. Server enables 2FA for the user.

On subsequent sign-ins:

1. User submits email and password to `POST /api/auth/login`.
2. Server returns `{ ok: true, pending: true, pluginId: 'twofa', userId }` instead of session tokens.
3. Client prompts for a six-digit code.
4. Client calls `POST /api/auth/2fa/verify` with `{ code }`.
5. Server validates the code and returns session tokens.

<Callout type="info">
  The plugin automatically hooks into the sign-in flow when 2FA is enabled. No additional middleware is required.
</Callout>
</Step>
</Steps>

## How it works

When a user with 2FA enabled signs in with a valid password:

1. Core sign-in flow detects the `twofa` plugin is registered and that the user has 2FA enabled.
2. Instead of issuing session tokens, core issues a short-lived pending token and calls the plugin `challenge` hook.
3. The pending token is stored in an `httpOnly` cookie.
4. The client receives `{ ok: true, pending: true, pluginId: 'twofa' }` and must call the 2FA verify endpoint.
5. The verify endpoint validates the TOTP code against the stored secret, consumes the pending token, and issues full session tokens.

See [Sessions](/concepts/sessions) for details on pending tokens and the challenge flow.
