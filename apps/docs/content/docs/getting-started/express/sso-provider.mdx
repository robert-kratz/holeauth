# Become an OIDC provider

This guide walks you through setting up your Express application as an OAuth 2.0 and OpenID Connect identity provider using `@holeauth/plugin-idp`. Your application will issue access tokens and ID tokens to other applications, expose an OIDC discovery endpoint, serve a JWKS, and support refresh-token rotation.

<Callout type="info">
  This is the **provider** (server) side — your app issues tokens to other apps. If you want to **consume** an external identity provider (sign in with Google, GitHub, etc.), see [Sign in with Google, GitHub, OIDC](/getting-started/express/sso-consumer).
</Callout>

<Callout type="info">
  This guide assumes you have completed the [core Express setup](/getting-started/express).
</Callout>

The IDP schema includes tables for:

- `teams` — multi-tenant app ownership
- `team_members` — users with `owner` or `developer` roles per team
- `apps` — OAuth 2.0 client registrations (public / confidential, allowed scopes, redirect URIs, PKCE policy)
- `authorization_codes` — short-lived codes issued during the authorization flow
- `refresh_tokens` — opaque refresh tokens with family-based rotation
- `consents` — remembered user × app × scope-set consent decisions
- `signing_keys` — JWKs for RS256 / ES256 token signing with automatic rotation

Run `drizzle-kit generate` and `drizzle-kit push` to apply the migration.

<Step>
### Wire the plugin into your auth instance

<Tabs items={['Postgres', 'MySQL', 'SQLite']}>
  <Tab value="Postgres">
```ts title="lib/auth.ts" {4,5,9,23-29}
import { createExpressAuth } from '@holeauth/express';
import { createHoleauthAdapters } from '@holeauth/adapter-drizzle/pg';
import { db } from '../db/client';
import { holeauth, idpSchema } from '../db/schema';
import { idp } from '@holeauth/plugin-idp';
import { createIdpAdapter } from '@holeauth/idp-drizzle/pg';

const adapters = createHoleauthAdapters({ db, tables: holeauth.tables });
const idpAdapter = createIdpAdapter({ db, tables: idpSchema.tables });

export const auth = createExpressAuth({
  secrets: {
    jwtSecret: process.env.HOLEAUTH_SECRET!,
  },
  adapters: {
    user: adapters.user,
    session: adapters.session,
    account: adapters.account,
    auditLog: adapters.auditLog,
    verificationToken: adapters.verificationToken,
    transaction: adapters.transaction,
  },
  plugins: [
    idp({
      adapter: idpAdapter,
      issuer: `${process.env.APP_URL}/api/auth`,
      scopesSupported: ['openid', 'profile', 'email', 'offline_access'],
      signingAlg: 'RS256',
    }),
  ],
  tokens: {
    accessTtl: 60 * 15,
    refreshTtl: 60 * 60 * 24 * 30,
    cookiePrefix: 'holeauth',
    sameSite: 'lax',
  },
  registration: {
    selfServe: true,
    inviteTtlSeconds: 7 * 24 * 60 * 60,
    inviteUrl: ({ token }) =>
      `${process.env.APP_URL}/register/accept?token=${encodeURIComponent(token)}`,
  },
  allowDangerousEmailAccountLinking: false,
});
```

<Step>
### Register an OAuth client app

To allow other applications to use your IDP, register them as OAuth clients. You can do this programmatically via the IDP adapter or through a management UI you build.

Example: registering a client via the adapter:

```ts title="scripts/register-client.ts"
import { db } from '../db/client';
import { idpSchema } from '../db/schema';
import { createIdpAdapter } from '@holeauth/idp-drizzle/pg';

const adapter = createIdpAdapter({ db, tables: idpSchema.tables });

async function main() {
  // Create a team (required for app ownership)
  const team = await adapter.createTeam({
    name: 'My Team',
    ownerId: 'user-id-of-owner',
  });

  // Register the app
  const app = await adapter.createApp({
    teamId: team.id,
    name: 'My Client App',
    clientId: 'my-client-app',
    clientSecret: 'super-secret', // Store hashed in production
    redirectUris: ['http://localhost:4000/callback'],
    allowedScopes: ['openid', 'profile', 'email'],
    confidential: true,
    pkcePolicy: 'optional',
  });

  console.log('Registered app:', app);
}

main();
```

<Callout type="warn">
  Store client secrets hashed, never in plain text. Use bcrypt or argon2 to hash the secret before persisting it.
</Callout>
</Step>

## How it works

When the IDP plugin is registered:

1. Holeauth generates an RSA or ECDSA key pair (depending on `signingAlg`) and stores it in the `signing_keys` table.
2. The OIDC discovery document is served at `/.well-known/openid-configuration`.
3. The JWKS (public keys) is served at `/.well-known/jwks.json`.
4. When a client app redirects a user to `/api/auth/authorize?client_id=...&redirect_uri=...&response_type=code&scope=openid`:
   - Holeauth checks if the user is authenticated. If not, redirects to your login page with a `returnTo` parameter.
   - If authenticated, checks the `consents` table for a prior consent decision.
   - If no consent exists, shows a consent screen (you must build this UI).
   - Once consented, issues an authorization code and redirects back to the client app `redirect_uri`.
5. The client app exchanges the code for tokens at `POST /api/auth/token`.
6. Holeauth validates the code, issues an access token (JWT) and ID token (JWT), and optionally a refresh token (opaque).
7. The client app can fetch user profile data from `/api/auth/userinfo` using the access token.

See [OIDC Specification](https://openid.net/specs/openid-connect-core-1_0.html) for the full protocol details.

## Building a consent screen

Holeauth does not provide a consent screen UI. You must build one yourself. When a user is redirected to `/api/auth/authorize` and no consent exists, holeauth will redirect to your consent URL (configurable via the `consentUrl` option in the plugin config) with query parameters:

- `client_id` — the OAuth client requesting consent
- `scope` — space-separated scopes requested
- `state` — CSRF token (must be echoed back)
- `redirect_uri` — where to redirect after consent

Your consent screen should:

1. Display the app name and requested scopes.
2. Prompt the user to allow or deny.
3. On allow, call `POST /api/auth/consent` with `{ clientId, scope, state }` (you will need to implement this endpoint or use the IDP API directly).
4. Redirect back to the authorization endpoint with the `state` parameter.

See [`@holeauth/plugin-idp`](/packages/plugin-idp) for the full consent flow API.
