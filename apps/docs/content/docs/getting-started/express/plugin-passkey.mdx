# Passkeys

This guide walks you through adding WebAuthn-based passkey authentication to your Express application. Users will register a passkey using their device biometrics or a hardware security key, then sign in without a password.

<Callout type="info">
  This guide assumes you have completed the [core Express setup](/getting-started/express).
</Callout>

Import the passkey table factory and add it to your schema:

<Tabs items={['Postgres', 'MySQL', 'SQLite']}>
  <Tab value="Postgres">
```ts title="db/schema.ts" {3,13-14}
import { pgTable, text, timestamp } from 'drizzle-orm/pg-core';
import { createHoleauthTables } from '@holeauth/adapter-drizzle/pg';
import { createPasskeyTables } from '@holeauth/passkey-drizzle/pg';

export const users = pgTable('user', {
  id: text('id').primaryKey().$defaultFn(() => crypto.randomUUID()),
  email: text('email').notNull().unique(),
  passwordHash: text('password_hash'),
  createdAt: timestamp('created_at', { withTimezone: true, mode: 'date' }).notNull().defaultNow(),
});

export const holeauth = createHoleauthTables({ usersTable: users });
export const passkeys = createPasskeyTables({ usersTable: users });
```

<Step>
### Wire the plugin into your auth instance

<Tabs items={['Postgres', 'MySQL', 'SQLite']}>
  <Tab value="Postgres">
```ts title="lib/auth.ts" {4,5,9,23-29}
import { createExpressAuth } from '@holeauth/express';
import { createHoleauthAdapters } from '@holeauth/adapter-drizzle/pg';
import { db } from '../db/client';
import { holeauth, passkeys } from '../db/schema';
import { passkey } from '@holeauth/plugin-passkey';
import { createPasskeyAdapter } from '@holeauth/passkey-drizzle/pg';

const adapters = createHoleauthAdapters({ db, tables: holeauth.tables });
const passkeyAdapter = createPasskeyAdapter({ db, tables: passkeys.tables });

export const auth = createExpressAuth({
  secrets: {
    jwtSecret: process.env.HOLEAUTH_SECRET!,
  },
  adapters: {
    user: adapters.user,
    session: adapters.session,
    account: adapters.account,
    auditLog: adapters.auditLog,
    verificationToken: adapters.verificationToken,
    transaction: adapters.transaction,
  },
  plugins: [
    passkey({
      adapter: passkeyAdapter,
      rpID: process.env.PASSKEY_RP_ID ?? 'localhost',
      rpName: 'MyApp',
      rpOrigin: process.env.APP_URL ?? 'http://localhost:3000',
    }),
  ],
  tokens: {
    accessTtl: 60 * 15,
    refreshTtl: 60 * 60 * 24 * 30,
    cookiePrefix: 'holeauth',
    sameSite: 'lax',
  },
  registration: {
    selfServe: true,
    inviteTtlSeconds: 7 * 24 * 60 * 60,
    inviteUrl: ({ token }) =>
      `${process.env.APP_URL}/register/accept?token=${encodeURIComponent(token)}`,
  },
  allowDangerousEmailAccountLinking: false,
});
```

<Step>
### Client integration

Your client must use the WebAuthn JavaScript APIs to perform passkey ceremonies. A typical registration flow:

1. User signs in and navigates to account settings.
2. Client calls `POST /api/auth/passkey/register/options` (send CSRF token).
3. Server returns `{ options: PublicKeyCredentialCreationOptions }`.
4. Client calls `navigator.credentials.create({ publicKey: options })`.
5. Browser prompts for biometric or security key.
6. Client calls `POST /api/auth/passkey/register/verify` with `{ response: credential, deviceName: 'My Phone' }`.
7. Server stores the credential.

A typical login flow:

1. Client calls `POST /api/auth/passkey/login/options`.
2. Server returns `{ options: PublicKeyCredentialRequestOptions }`.
3. Client calls `navigator.credentials.get({ publicKey: options })`.
4. Browser prompts for biometric or security key.
5. Client calls `POST /api/auth/passkey/login/verify` with `{ response: credential }`.
6. Server validates the signature and returns session tokens.

<Callout type="info">
  The `@simplewebauthn/browser` package provides `startRegistration` and `startAuthentication` helpers that handle the WebAuthn API calls and response serialization. Install it in your client: `pnpm add @simplewebauthn/browser`.
</Callout>
</Step>

## How it works

Passkeys use public-key cryptography. During registration:

1. The server generates a random challenge and sends it to the client along with the RP ID and user identifier.
2. The client calls `navigator.credentials.create()`, which prompts the user for biometric authentication or a security key.
3. The authenticator generates a key pair. The private key stays on the device; the public key is sent to the server.
4. The server stores the public key, credential ID, and device metadata.

During login:

1. The server generates a challenge and sends it to the client.
2. The client calls `navigator.credentials.get()`, which prompts the user to authenticate.
3. The authenticator signs the challenge with the private key and returns a signed assertion.
4. The server verifies the signature using the stored public key.
5. If valid, the server issues session tokens.

See [WebAuthn Guide](https://webauthn.guide) for a detailed explanation of the WebAuthn protocol.
