# Become an OIDC provider

This guide walks you through setting up your Next.js App Router application as an OAuth 2.0 and OpenID Connect identity provider using `@holeauth/plugin-idp`. Your application will issue access tokens and ID tokens to other applications, expose an OIDC discovery endpoint, serve a JWKS, and support refresh-token rotation.

<Callout type="info">
  This is the **provider** (server) side — your app issues tokens to other apps. If you want to **consume** an external identity provider (sign in with Google, GitHub, etc.), see [Sign in with Google, GitHub, OIDC](/getting-started/nextjs-app-router/sso-consumer).
</Callout>

<Callout type="info">
  This guide assumes you have completed the [core Next.js App Router setup](/getting-started/nextjs-app-router).
</Callout>

The IDP schema includes tables for:

- `teams` — multi-tenant app ownership
- `team_members` — users with `owner` or `developer` roles per team
- `apps` — OAuth 2.0 client registrations (public / confidential, allowed scopes, redirect URIs, PKCE policy)
- `authorization_codes` — short-lived codes issued during the authorization flow
- `refresh_tokens` — opaque refresh tokens with family-based rotation
- `consents` — remembered user × app × scope-set consent decisions
- `signing_keys` — JWKs for RS256 / ES256 token signing with automatic rotation

Run `drizzle-kit generate` and `drizzle-kit push` to apply the migration.
</Step>

<Step>
### Wire the plugin into your auth instance

<Tabs items={['Postgres', 'MySQL', 'SQLite']}>
  <Tab value="Postgres">
```ts title="lib/auth.ts" {4,5,9,23-29}
import { createAuthHandler } from '@holeauth/nextjs-app-router';
import { createHoleauthAdapters } from '@holeauth/adapter-drizzle/pg';
import { db } from '@/db/client';
import { holeauth, idpSchema } from '@/db/schema';
import { idp } from '@holeauth/plugin-idp';
import { createIdpAdapter } from '@holeauth/idp-drizzle/pg';

const adapters = createHoleauthAdapters({ db, tables: holeauth.tables });
const idpAdapter = createIdpAdapter({ db, tables: idpSchema.tables });

export const auth = createAuthHandler({
  secrets: {
    jwtSecret: process.env.HOLEAUTH_SECRET!,
  },
  adapters: {
    user: adapters.user,
    session: adapters.session,
    account: adapters.account,
    auditLog: adapters.auditLog,
    verificationToken: adapters.verificationToken,
    transaction: adapters.transaction,
  },
  plugins: [
    idp({
      adapter: idpAdapter,
      issuer: `${process.env.APP_URL}/api/auth`,
      scopesSupported: ['openid', 'profile', 'email', 'offline_access'],
      signingAlg: 'RS256',
    }),
  ],
  tokens: {
    accessTtl: 60 * 15,
    refreshTtl: 60 * 60 * 24 * 30,
    cookiePrefix: 'holeauth',
    sameSite: 'lax',
  },
  registration: {
    selfServe: true,
    inviteTtlSeconds: 7 * 24 * 60 * 60,
    inviteUrl: ({ token }) =>
      `${process.env.APP_URL}/register/accept?token=${encodeURIComponent(token)}`,
  },
  allowDangerousEmailAccountLinking: false,
});
```

<Step>
### Register a client application

Use the plugin API to create an OAuth 2.0 client:

```ts title="app/actions/idp.ts"
'use server';
import { auth } from '@/lib/auth';

export async function createOAuthApp(name: string, redirectUris: string[]) {
  const session = await auth.getSession();
  if (!session) throw new Error('Unauthorized');

  // Check permission (requires plugin-rbac)
  const canCreate = await auth.rbac?.can(session.userId, 'idp.apps.create');
  if (!canCreate) throw new Error('Forbidden');

  const app = await auth.idp.createApp({
    name,
    teamId: session.userId, // or a real team ID from your schema
    redirectUris,
    type: 'confidential', // or 'public' for mobile/SPA apps
    allowedScopes: ['openid', 'profile', 'email'],
    requirePkce: true,
  });

  return { clientId: app.id, clientSecret: app.clientSecret };
}
```

<Callout type="warn">
  Store the `clientSecret` securely. It is hashed in the database and cannot be retrieved later. Display it to the user exactly once during app creation.
</Callout>
</Step>

<Step>
### Discovery and JWKS endpoints

The plugin automatically exposes the following routes:

- `GET /api/auth/.well-known/openid-configuration` — OIDC discovery document
- `GET /api/auth/oauth2/jwks` — JSON Web Key Set (public keys for token verification)
- `GET /api/auth/oauth2/authorize` — authorization endpoint (initiates the OAuth flow)
- `POST /api/auth/oauth2/token` — token endpoint (exchanges code for tokens, refreshes tokens)
- `POST /api/auth/oauth2/revoke` — revocation endpoint (RFC 7009)
- `GET /api/auth/oauth2/userinfo` — userinfo endpoint (returns claims for the current access token)
- `POST /api/auth/oauth2/end-session` — RP-initiated logout endpoint

Clients can discover your provider by fetching `https://your-domain.com/api/auth/.well-known/openid-configuration`.
</Step>

<Step>
### Authorization flow walkthrough

1. A Relying Party redirects the user to `/api/auth/oauth2/authorize?client_id=...&redirect_uri=...&response_type=code&scope=openid profile email&state=...&code_challenge=...&code_challenge_method=S256`.
2. The user signs in to your app (if not already authenticated).
3. Your app displays a consent screen asking the user to approve the requested scopes.
4. On approval, your app redirects to the RP callback URL with an authorization code: `https://rp.example.com/callback?code=...&state=...`.
5. The RP POSTs the code to `/api/auth/oauth2/token` with its client credentials and PKCE verifier.
6. Your provider verifies the code and PKCE, then issues an access token, ID token, and refresh token.
7. The RP can now call `/api/auth/oauth2/userinfo` with the access token to fetch user claims.

See the [OAuth 2.0 Authorization Code Flow](https://oauth.net/2/grant-types/authorization-code/) and [OIDC Core spec](https://openid.net/specs/openid-connect-core-1_0.html) for detailed protocol documentation.
</Step>

<Step>
### Token revocation

Clients can revoke access tokens or refresh tokens by POSTing to `/api/auth/oauth2/revoke`:

```http
POST /api/auth/oauth2/revoke HTTP/1.1
Host: your-domain.com
Content-Type: application/x-www-form-urlencoded
Authorization: Basic {base64(clientId:clientSecret)}

token={refresh_token}&token_type_hint=refresh_token
```

Revoking a refresh token revokes all tokens in the same family (reuse detection).
</Step>

<Step>
### Multi-tenant apps (teams)

The IDP plugin supports team-based app ownership. Each OAuth app belongs to a team. Users can be team members with `owner` or `developer` roles:

- `owner` — can delete the app, manage team members, regenerate client secrets.
- `developer` — can edit app settings (redirect URIs, scopes), view client credentials.

Create a team and add members:

```ts
await auth.idp.createTeam({ name: 'Acme Corp', ownerId: userId });
await auth.idp.addTeamMember({ teamId, userId, role: 'developer' });
```

Apps are created within a team:

```ts
await auth.idp.createApp({ name: 'Acme Dashboard', teamId, ... });
```

This allows multi-tenant SaaS applications to isolate OAuth apps per customer organization.
</Step>
</Steps>

## How it works

1. The plugin generates or loads RSA / ECDSA key pairs for token signing and stores them in the `signing_keys` table. Keys rotate automatically based on the `signingAlg` and key lifetime policy.
2. Access tokens are JWTs signed with the active key. They contain standard OIDC claims (`iss`, `sub`, `aud`, `exp`, `iat`, `scope`).
3. Refresh tokens are opaque UUIDs hashed with SHA-256 and stored in the `refresh_tokens` table. Each refresh has a `familyId`. On rotation, the old token is marked revoked and a new token with the same `familyId` is issued. If an already-revoked token is reused, the entire family is revoked (refresh-token reuse detection).
4. Authorization codes are short-lived (default 10 minutes), single-use, and bound to a PKCE challenge.
5. User consent decisions are stored in the `consents` table keyed by `userId`, `appId`, and `scopeHash`. If the user has already consented to the exact scope set, the consent screen is skipped on subsequent authorizations.

This design matches the security model of `@holeauth/core/session` for the IDP's own sessions — refresh-token rotation with family-based revocation is the same pattern used for user sessions in your app.
