# Events and Audit Log

Every authentication flow in holeauth emits typed events that you can subscribe to for logging, alerting, or analytics. Events are persisted to the database via the `AuditLogAdapter` and can be queried later for compliance or debugging.

## Event System

Events are published using an internal event emitter. Each event has a `type` string and carries metadata like `userId`, `sessionId`, `ip`, and `userAgent`. Plugins can emit their own events under namespaced type strings (for example, `twofa.verified` or `rbac.group_assigned`).

Subscribe to events using the `auth.on` method:

```ts title="lib/auth.ts"
import { defineHoleauth } from '@holeauth/core';
import { drizzleAdapter } from '@holeauth/adapter-drizzle/pg';
import { db } from '@/db';

export const auth = defineHoleauth({
  adapter: drizzleAdapter(db),
  secrets: { jwtSecret: process.env.HOLEAUTH_SECRET! },
});

auth.on('user.registered', async (event) => {
  console.log('New user registered:', event.userId);
  await sendWelcomeEmail(event.userId);
});

auth.on('session.reuse_detected', async (event) => {
  console.error('Session hijack attempt:', event.data);
  await sendSecurityAlert(event.userId, 'Suspicious activity detected');
});
```

The handler is called asynchronously and errors are swallowed to avoid blocking the authentication flow.

## Core Event Types

holeauth emits 19 well-known event types:

| Event Type | Description |
| --- | --- |
| `user.registered` | A new user account was created via `auth.register`. |
| `user.signed_in` | A user signed in successfully. |
| `user.signed_out` | A user signed out (session revoked). |
| `user.updated` | User profile was updated via `auth.updateUser`. |
| `user.deleted` | User account was deleted via `auth.deleteUser`. |
| `session.created` | A new session was issued (login, 2FA verify, SSO callback). |
| `session.rotated` | A refresh token was rotated via `auth.refresh`. |
| `session.revoked` | A session was explicitly revoked (signout, reuse, expiration). |
| `session.reuse_detected` | A revoked refresh token was reused; family revoked. |
| `password.changed` | User changed their password via `auth.changePassword`. |
| `password.reset_requested` | User requested a password reset link. |
| `password.reset_consumed` | User completed password reset via the token link. |
| `account.linked` | An OAuth/OIDC account was linked to a user. |
| `account.unlinked` | An OAuth/OIDC account was unlinked from a user. |
| `sso.authorize` | User initiated an OAuth/OIDC authorization flow. |
| `sso.callback_ok` | OAuth/OIDC callback succeeded. |
| `sso.callback_failed` | OAuth/OIDC callback failed (invalid state, code exchange error). |
| `invite.created` | An invite token was created via `auth.createInvite`. |
| `user.invite_consumed` | A user registered via an invite token. |
| `invite.revoked` | An invite token was revoked via `auth.revokeInvite`. |
| `plugin.error` | A plugin hook threw an error. |

<Callout type="info">
  This list represents the core events. Plugins emit additional events under their own namespaces. For example, the RBAC plugin emits `rbac.group_assigned` and `rbac.permission_checked`.
</Callout>

## Event Payload

All events extend the `AdapterAuditEvent` interface:

    },
    type: {

      type: 'string',

    },
    userId: {
      description: 'ID of the user who triggered the event, if applicable.',
      type: 'string | null',
      required: false,
    },
    sessionId: {
      description: 'ID of the session involved, if applicable.',
      type: 'string | null',
      required: false,
    },
    at: {
      description: 'Timestamp of the event (ISO 8601).',

      required: false,
    },
    ip: {
      description: 'IP address of the client, if available.',
      type: 'string | null',
      required: false,
    },
    userAgent: {
      description: 'User-Agent header of the client, if available.',
      type: 'string | null',
      required: false,
    },
    data: {
      description: 'Opaque JSON payload with event-specific details.',
      type: 'Record<string, unknown> | null',
      required: false,
    },
  }}
/>

The `data` field varies by event type. For example:

- `session.reuse_detected` includes `{ familyId: string }`.
- `sso.callback_ok` includes `{ provider: string, providerAccountId: string }`.
- `user.registered` includes `{ email: string, name?: string }`.

## Subscribing to Events

Use the `auth.on` method to register a handler:

```ts title="lib/auth.ts"
const unsubscribe = auth.on('user.signed_in', async (event) => {
  console.log(`User ${event.userId} signed in from ${event.ip}`);
});
```

The method returns an unsubscribe function. Call it to remove the handler:

```ts
unsubscribe();
```

You can subscribe to the same event type multiple times. All handlers are called in the order they were registered.

## Legacy onEvent Hook

For backward compatibility, holeauth still supports the `onEvent` configuration option:

```ts title="lib/auth.ts"
export const auth = defineHoleauth({
  adapter: drizzleAdapter(db),
  secrets: { jwtSecret: process.env.HOLEAUTH_SECRET! },
  onEvent: async (event) => {
    console.log('Event:', event.type);
  },
});
```

This is called for every event. New code should use `auth.on` instead.

## Audit Log Adapter

All events are persisted to the database via the `AuditLogAdapter`:

```ts title="@holeauth/core/adapters"
export interface AuditLogAdapter {
  record(event: AdapterAuditEvent): Promise<void>;
  list?(filter: {
    userId?: string;
    type?: string;
    limit?: number;
  }): Promise<AdapterAuditEvent[]>;
}
```

The `record` method is mandatory and must be awaited by every holeauth flow. The `list` method is optional and used for querying the audit log.

<Callout type="warn">
  Do not skip persisting events. The audit log is your forensic trail for debugging authentication issues and satisfying compliance requirements.
</Callout>

Example Drizzle implementation:

```ts title="lib/adapters/audit.ts"
import { db, auditLog } from '@/db';
import type { AuditLogAdapter, AdapterAuditEvent } from '@holeauth/core/adapters';

export const auditLogAdapter: AuditLogAdapter = {
  async record(event: AdapterAuditEvent) {
    await db.insert(auditLog).values({
      type: event.type,
      userId: event.userId ?? null,
      sessionId: event.sessionId ?? null,
      at: event.at ?? new Date(),
      ip: event.ip ?? null,
      userAgent: event.userAgent ?? null,
      data: event.data ?? null,
    });
  },
  async list(filter) {
    const rows = await db
      .select()
      .from(auditLog)
      .where(filter.userId ? eq(auditLog.userId, filter.userId) : undefined)
      .orderBy(desc(auditLog.at))
      .limit(filter.limit ?? 100);
    return rows;
  },
};
```

## Plugin Events

Plugins emit their own events under namespaced type strings. The naming convention is `<pluginId>.<event>`. For example:

- `twofa.setup`
- `twofa.verified`
- `passkey.registered`
- `passkey.used`
- `rbac.group_assigned`
- `rbac.group_removed`

Subscribe to plugin events the same way:

```ts title="lib/auth.ts"
auth.on('twofa.verified', async (event) => {
  console.log('2FA challenge passed:', event.userId);
});
```

<Callout type="info">
  Plugin event types are not included in the `CoreHoleauthEventType` union, but they are accepted via the `(string & {})` escape hatch. Your IDE will not autocomplete them.
</Callout>
