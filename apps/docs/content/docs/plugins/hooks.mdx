# Hooks

Plugins register hook functions that run before or after core flows. Hooks are the primary mechanism for plugins to observe and influence authentication events.

## Before vs. after semantics

### Before hooks

`before` hooks run **before** the core flow executes. They may throw to abort the flow. Exceptions propagate to the caller (HTTP dispatcher or application code) and prevent the operation from proceeding.

**Use cases:** Validation, rate limiting, pre-flight checks, custom authorization.

```ts
import { definePlugin, type PluginContext } from '@holeauth/core/plugins';

export const rateLimiter = () => definePlugin({
  id: 'rateLimiter' as const,
  hooks: {
    signIn: {
      async before(input, ctx) {
        const attempts = await checkAttempts(input.email);
        if (attempts > 5) {
          throw new Error('Rate limit exceeded');
        }
      },
    },
  },
  api: () => ({}),
});
```

### After hooks

`after` hooks run **after** the core flow has succeeded. They **cannot abort** the flow. Exceptions from `after` hooks are caught by core, logged via `ctx.logger`, and emitted as `plugin.error` events. The flow continues normally.

**Use cases:** Audit logging, analytics, notifications, cache invalidation, downstream side effects.

```ts
import { definePlugin, type PluginContext } from '@holeauth/core/plugins';

export const auditLogger = () => definePlugin({
  id: 'auditLogger' as const,
  hooks: {
    signIn: {
      async after(result, ctx) {
        await ctx.events.emit({
          type: 'audit.login',
          userId: result.user.id,
          data: { method: result.method, timestamp: Date.now() },
        });
      },
    },
  },
  api: () => ({}),
});
```

<Callout type="warn">
  **Never throw from `after` hooks.** Core will catch the exception and emit a `plugin.error` event, but the flow has already completed. If you need to abort a flow, throw from a `before` hook.
</Callout>

## Challenge hooks

The `signIn.challenge` hook is special: it returns a `ChallengeResult | null` to optionally halt sign-in and require a follow-up verification step (e.g. TOTP code, passkey ceremony, magic link click).

**Semantics:**

- If no plugin returns a `ChallengeResult`, sign-in proceeds normally.
- If one or more plugins return a `ChallengeResult`, only the **first non-null** wins. Subsequent non-null results trigger a warning and are ignored.
- The winning plugin's `pluginId` is set in the response, and a short-lived `pendingToken` (JWT) is issued.
- The client must drive the plugin's follow-up route (e.g. `POST /2fa/verify`) with the pending token + verification data.

```ts
import { definePlugin, type ChallengeResult, type PluginContext } from '@holeauth/core/plugins';
import { issuePendingToken } from '@holeauth/core/flows';

export const twofa = () => definePlugin({
  id: 'twofa' as const,
  hooks: {
    signIn: {
      async challenge(user, input, ctx): Promise<ChallengeResult | null> {
        const record = await ctx.getPluginAdapter<TwoFactorAdapter>('twofa')?.getByUserId(user.id);
        if (!record?.enabled) return null;

        const { token, expiresAt } = await issuePendingToken(ctx.config, {
          userId: user.id,
          pluginId: 'twofa',
          ttlSeconds: 300,
        });

        return {
          pluginId: 'twofa',
          pendingToken: token,
          expiresAt,
        };
      },
    },
  },
  api: () => ({}),
});
```

## Hook reference

### register

| Slot | Arguments | Returns | Semantics |
| --- | --- | --- | --- |
| `before` | `RegisterHookInput, PluginContext` | `Promise<void> \| void` | May throw to abort. |
| `after` | `AdapterUser, PluginContext` | `Promise<void> \| void` | Non-destructive. |

<TypeTable
  type={{
    email: {
      description: 'Email address being registered.',
      type: 'string',
      required: true,
    },
    password: {
      description: 'Plaintext password (hashed by core before persisting).',
      type: 'string',
      required: true,
    },
    name: {
      description: 'Optional display name.',
      type: 'string | null',
    },
  }}
/>

**Example:**

```ts
hooks: {
  register: {
    async before(input, ctx) {
      if (input.email.endsWith('@blocked.com')) {
        throw new Error('Domain blocked');
      }
    },
    async after(user, ctx) {
      await ctx.events.emit({
        type: 'analytics.user_created',
        userId: user.id,
        data: { email: user.email },
      });
    },
  },
}
```

```ts
hooks: {
  signIn: {
    async before(input, ctx) {
      const blocked = await ctx.getPluginAdapter<BlocklistAdapter>('blocklist')?.isBlocked(input.email);
      if (blocked) throw new Error('Email blocked');
    },
    async challenge(user, input, ctx) {
      const twofa = await ctx.getPluginAdapter<TwoFactorAdapter>('twofa')?.getByUserId(user.id);
      if (!twofa?.enabled) return null;
      const { token, expiresAt } = await issuePendingToken(ctx.config, {
        userId: user.id,
        pluginId: 'twofa',
        ttlSeconds: 300,
      });
      return { pluginId: 'twofa', pendingToken: token, expiresAt };
    },
    async after(result, ctx) {
      ctx.logger.info(`User ${result.user.id} signed in via ${result.method}`);
    },
  },
}
```

## Hook error handling

Exceptions from `after` hooks are caught by core and emitted as `plugin.error` events. The flow continues normally.

The `plugin.error` event shape:

```ts
{
  type: 'plugin.error',
  data: {
    hook: string,       // e.g. 'signIn.after'
    error: string,      // Error message
  }
}
```

Subscribe to this event to surface plugin errors to an error tracking service:

```ts
auth.events.on('plugin.error', async (e) => {
  console.error('Plugin hook failed:', e.data.hook, e.data.error);
  await reportToSentry(e);
});
```

<Callout type="info">
  Core logs every `plugin.error` via `ctx.logger.error`. You do not need to log them again unless you want custom formatting.
</Callout>
