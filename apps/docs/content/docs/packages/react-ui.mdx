# @holeauth/react-ui

Headless, polymorphic UI components for holeauth. Provides pre-built forms for sign-in, sign-up, password reset, two-factor verification, and passkey management. All components are unstyled and accept an `as` prop for custom rendering.

## Quick example

```tsx title="app/login/page.tsx"
import { SignInForm } from '@holeauth/react-ui';

export default function LoginPage() {
  return (
    <SignInForm.Root onSuccess={() => window.location.href = '/dashboard'}>
      <SignInForm.Email placeholder="Email" />
      <SignInForm.Password placeholder="Password" />
      <SignInForm.Submit>Sign In</SignInForm.Submit>
      <SignInForm.Error />
    </SignInForm.Root>
  );
}
```

## Compound forms

### SignInForm

Sign-in form with email, password, passkey, and challenge (2FA) support.

```tsx
import { SignInForm } from '@holeauth/react-ui';

<SignInForm.Root onSuccess={(session) => console.log(session)}>
  <SignInForm.Email placeholder="Email" required />
  <SignInForm.Password placeholder="Password" required />
  <SignInForm.Submit>Sign In</SignInForm.Submit>
  <SignInForm.PasskeyButton>Sign in with Passkey</SignInForm.PasskeyButton>
  <SignInForm.Error />
  <SignInForm.Pending />
</SignInForm.Root>
```

<TypeTable
  type={{
    onSuccess: {
      description: 'Callback when sign-in completes.',
      type: '(session: ClientSession) => void',
    },
    onPendingChallenge: {
      description: 'Callback when a challenge (2FA) is required.',
      type: '(challenge: PendingChallenge) => void',
    },
  }}
/>

**Components:**

- `SignInForm.Root` — Form wrapper
- `SignInForm.Email` — Email input
- `SignInForm.Password` — Password input
- `SignInForm.Submit` — Submit button
- `SignInForm.PasskeyButton` — Passkey login button
- `SignInForm.Error` — Error message display
- `SignInForm.Pending` — Pending challenge message

### SignUpForm

Sign-up form with email, password, and optional name.

```tsx
import { SignUpForm } from '@holeauth/react-ui';

<SignUpForm.Root onSuccess={() => console.log('Registered')}>
  <SignUpForm.Field name="email" type="email" placeholder="Email" required />
  <SignUpForm.Field name="name" placeholder="Name" />
  <SignUpForm.Field name="password" type="password" placeholder="Password" required />
  <SignUpForm.Submit>Sign Up</SignUpForm.Submit>
  <SignUpForm.Error />
</SignUpForm.Root>
```

### PasswordResetRequestForm

Password reset request form (sends email).

```tsx
import { PasswordResetRequestForm } from '@holeauth/react-ui';

<PasswordResetRequestForm.Root onSuccess={() => console.log('Email sent')}>
  <PasswordResetRequestForm.Email placeholder="Email" required />
  <PasswordResetRequestForm.Submit>Send Reset Link</PasswordResetRequestForm.Submit>
  <PasswordResetRequestForm.Error />
  <PasswordResetRequestForm.Success />
</PasswordResetRequestForm.Root>
```

### PasswordChangeForm

Password change form (authenticated users).

```tsx
import { PasswordChangeForm } from '@holeauth/react-ui';

<PasswordChangeForm.Root onSuccess={() => console.log('Password changed')}>
  <PasswordChangeForm.Field name="currentPassword" type="password" placeholder="Current Password" required />
  <PasswordChangeForm.Field name="newPassword" type="password" placeholder="New Password" required />
  <PasswordChangeForm.Submit>Change Password</PasswordChangeForm.Submit>
  <PasswordChangeForm.Error />
  <PasswordChangeForm.Success />
</PasswordChangeForm.Root>
```

### TwoFactorVerifyForm

Two-factor verification form. Reads the `holeauth.pending` cookie automatically — no `pendingToken` prop needed. Render this on a dedicated `/2fa/verify` page; do **not** embed it inline in the login page.

```tsx
import { TwoFactorVerifyForm } from '@holeauth/react-ui';

<TwoFactorVerifyForm.Root
  onSuccess={() => router.push('/dashboard')}
  onCancel={() => router.push('/login')}
>
  <TwoFactorVerifyForm.Code placeholder="6-digit code" required />
  <TwoFactorVerifyForm.Submit>Verify</TwoFactorVerifyForm.Submit>
  <TwoFactorVerifyForm.Error />
  <TwoFactorVerifyForm.Cancel>← Back to login</TwoFactorVerifyForm.Cancel>
</TwoFactorVerifyForm.Root>
```

`TwoFactorVerifyForm.Cancel` calls `POST /api/auth/2fa/cancel` to clear the pending cookie server-side, then triggers `onCancel`.

### PasskeySetup

Passkey management component (list, register, delete).

```tsx
import { PasskeySetup } from '@holeauth/react-ui';

<PasskeySetup.Root>
  <PasskeySetup.List />
  <PasskeySetup.RegisterButton>Add Passkey</PasskeySetup.RegisterButton>
  <PasskeySetup.DeleteButton credentialId="...">Delete</PasskeySetup.DeleteButton>
  <PasskeySetup.Error />
</PasskeySetup.Root>
```

## Polymorphic props

All components accept an `as` prop to render as a custom element.

```tsx
import { SignInForm } from '@holeauth/react-ui';

<SignInForm.Root as="div" className="custom-form">
  <SignInForm.Email as="input" className="custom-input" />
  <SignInForm.Submit as="button" className="custom-button">
    Sign In
  </SignInForm.Submit>
</SignInForm.Root>
```

## Hook access

Every compound form exposes its state via a custom hook.

```tsx
import { useSignInForm } from '@holeauth/react-ui';

function CustomSignInForm() {
  const form = useSignInForm({
    onSuccess: (session) => console.log(session),
  });

  return (
    <form onSubmit={form.handleSubmit}>
      <input {...form.emailProps} />
      <input {...form.passwordProps} />
      <button type="submit" disabled={form.loading}>
        Sign In
      </button>
      {form.error && <p>{form.error.message}</p>}
    </form>
  );
}
```

## Styling and Theming

`@holeauth/react-ui` ships unstyled. Every primitive renders a semantic element with a stable `data-holeauth-*` attribute, leaving visual decisions to your design system. Three layered approaches cover the common cases.

### 1. Per-component `className`

Pass `className` to any compound primitive or atomic button. The class is forwarded to the rendered element verbatim.

```tsx
<SignInForm.Root className="space-y-4 rounded-2xl border p-6">
  <SignInForm.Email className="w-full rounded-md border px-3 py-2" />
  <SignInForm.Password className="w-full rounded-md border px-3 py-2" />
  <SignInForm.Submit className="w-full rounded-md bg-black px-4 py-2 text-white">
    Sign in
  </SignInForm.Submit>
</SignInForm.Root>
```

### 2. Global CSS via `data-*` selectors

Each primitive emits a deterministic data attribute. Target those attributes from global CSS, Tailwind plugins, or design-token files to theme every form at once.

| Component | Root attribute | State attributes |
| --- | --- | --- |
| `SignInForm.Root` | `data-holeauth-signin-form` | \u2014 |
| `SignInForm.Email` | `data-holeauth-signin-email` | \u2014 |
| `SignInForm.Password` | `data-holeauth-signin-password` | \u2014 |
| `SignInForm.Submit` | `data-holeauth-signin-submit` | `data-loading` |
| `SignInForm.Error` | `data-holeauth-signin-error` | \u2014 |
| `PasskeyLoginButton` | `data-holeauth-signin-passkey` | `data-loading` |
| `SignInForm.Pending` | `data-holeauth-signin-pending` | \u2014 |

```css title="app/globals.css"
[data-holeauth-signin-form] {\n  display: grid;\n  gap: 0.75rem;\n}\n[data-holeauth-signin-submit][data-loading] {\n  opacity: 0.6;\n  pointer-events: none;\n}
```

### 3. Headless escape hatch

When the compound API does not match a particular design (multi-step wizard, modal layout, custom validation), drop down to the underlying hook. `useSignInForm`, `useSignUpForm`, and friends expose the same state machine that powers the compound primitives.

```tsx
import { useSignInForm } from '@holeauth/react-ui';

function CustomSignIn() {
  const form = useSignInForm();
  return (
    <form onSubmit={form.handleSubmit} className="my-custom-form">
      <MyTextField {...form.emailProps} label="Email" />
      <MyTextField {...form.passwordProps} label="Password" type="password" />
      <MyButton type="submit" loading={form.loading}>Continue</MyButton>
      {form.error && <MyAlert>{form.error.message}</MyAlert>}
    </form>
  );
}
```

## Accessibility

The primitives are built to pass WCAG 2.1 AA out of the box.

### ARIA wiring

- Each form root creates a stable `id` namespace; inputs reference the error region via `aria-describedby`.
- `*.Error` renders inside `role="alert"` (compound forms) so screen readers announce validation failures without focus loss.
- `*.Submit` and atomic buttons expose `aria-busy` while `data-loading` is set, mirroring async state.
- Compound forms set `noValidate` on the underlying `<form>` and surface field errors through the hook \u2014 the browser's native bubble UI never appears, so screen-reader output stays consistent across browsers.

### Keyboard navigation

| Key | Behaviour |
| --- | --- |
| `Tab` / `Shift+Tab` | Moves between fields and buttons in document order. The forms never use `tabindex` overrides. |
| `Enter` | Submits the form from any input. The same handler runs as clicking `*.Submit`. |
| `Space` / `Enter` on buttons | Activates atomic buttons (`PasskeyLoginButton`, `SignOutButton`, `SsoButton`). |
| `Escape` | No-op by default. Wrap a primitive in your own modal/dismiss logic to handle it. |

<Callout type="info">
  `PasskeyLoginButton` triggers the WebAuthn prompt on click, which the browser itself manages with full keyboard and assistive-technology support. The component only owns the trigger affordance.
</Callout>

### Focus management

The primitives never move focus implicitly. After a successful sign-in or sign-up, focus stays on the submit control until your `onSuccess` handler navigates away. Custom flows that need post-submit focus restoration should call `ref.focus()` from inside `onSuccess`.
