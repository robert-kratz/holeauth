# SSO Consumer

SSO consumer functionality is built into `@holeauth/core/sso`. There is no separate `@holeauth/plugin-idp-consumer` package. The framework adapters (Next.js, Express, Hono) automatically register the `/authorize/:provider` and `/callback/:provider` routes when SSO providers are configured.

<Callout type="info">
  This page documents the SSO consumer (acting as a Relying Party / OAuth client). To run an OAuth 2.0 / OIDC Identity Provider (SSO server), see [plugin-idp](/packages/plugin-idp).
</Callout>

## Installation

## Quick example

```ts title="lib/auth.ts"
import { defineHoleauth } from '@holeauth/core';
import { GoogleProvider, GithubProvider } from '@holeauth/core/sso';
import { drizzleAdapter } from '@holeauth/adapter-drizzle/pg';
import { db } from './db';

export const auth = defineHoleauth({
  adapter: drizzleAdapter(db),
  secrets: {
    jwtSecret: process.env.HOLEAUTH_SECRET!,
  },
  sso: {
    baseUrl: process.env.NEXT_PUBLIC_BASE_URL!,
    providers: [
      GoogleProvider({
        clientId: process.env.GOOGLE_CLIENT_ID!,
        clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
        redirectUri: `${process.env.NEXT_PUBLIC_BASE_URL}/api/auth/callback/google`,
      }),
      GithubProvider({
        clientId: process.env.GITHUB_CLIENT_ID!,
        clientSecret: process.env.GITHUB_CLIENT_SECRET!,
        redirectUri: `${process.env.NEXT_PUBLIC_BASE_URL}/api/auth/callback/github`,
      }),
    ],
  },
});
```

## Routes

Framework adapters automatically register these routes when SSO providers are configured:

- `GET /api/auth/authorize/:provider` — Start OAuth flow
- `GET /api/auth/callback/:provider` — Handle OAuth callback

## Client-side usage

### React

```tsx
import { useSso } from '@holeauth/react';

function GoogleSignInButton() {
  const { start } = useSso('google');

  return <button onClick={() => start('/dashboard')}>Sign in with Google</button>;
}
```

### React UI

```tsx
import { SsoButton } from '@holeauth/react-ui';

<SsoButton providerId="google">Sign in with Google</SsoButton>
```

## Server-side usage

```ts title="app/api/auth/sso/route.ts"
import { authorize, callback } from '@holeauth/core/sso';
import { auth } from '@/lib/auth';

// Start OAuth flow
const { redirectUrl, state, codeVerifier } = authorize('google', auth.sso);
// Store state and codeVerifier in a cookie
// Redirect to redirectUrl

// Handle callback
const result = await callback('google', {
  code: searchParams.get('code')!,
  state: searchParams.get('state')!,
  codeVerifier: cookies.get('holeauth.pkce'),
}, auth.sso);

// Find or create user with result.providerAccountId and result.email
```

## Account linking

When a user signs in via SSO, holeauth links the provider account to the user record. The `accounts` table stores:

- `provider` — Provider ID (e.g., `google`, `github`)
- `providerAccountId` — Provider's unique user ID
- `email` — User's email from the provider
- `accessToken` — Provider access token
- `refreshToken` — Provider refresh token (if available)
- `expiresAt` — Token expiration timestamp
- `tokenType` — Token type (e.g., `Bearer`)
- `scope` — Granted scopes
- `idToken` — OIDC ID token (if available)

## Security considerations

<Callout type="warn">
  - Always validate the `state` parameter to prevent CSRF attacks.
  - Use PKCE (Proof Key for Code Exchange) for public clients.
  - Rotate provider access tokens when they expire.
  - Revoke provider access when a user unlinks an account.
</Callout>
